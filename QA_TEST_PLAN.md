# خطة اختبار شاملة لمنصة بسهولة (Beshola)

تقرير اختبار كامل لكل وظيفة وميزة في النظام، مبني على فحص الكود الفعلي (Frontend + Supabase RLS + Edge Functions). الهدف: تغطية كل السيناريوهات (Happy Path + Edge Cases + Security) قبل الإطلاق.

---

## ⚠️ مشاكل أمنية حرجة لازم تتصلح قبل أي إطلاق (Critical Blockers)

هذول لازم يتعالجوا **قبل** ما تطلق، مش بعدها. باقي التقرير تيستات، لكن دول bugs/ثغرات لازم fix:

### C1 — ملفات المرضى (X-ray, تحاليل, تقارير) متاحة للعامة بدون أي حماية
- **الموقع:** `017_medical_files_public_read.sql` — bucket `medical-files` معمول `public` ومعاه policy تسمح SELECT لـ `anon`.
- **المشكلة:** أي شخص يعرف رابط الملف (أو يخمّن الـ path: `businessId/clientPhone/timestamp_random.ext`) يقدر يفتح صورة أشعة أو تحليل لمريض من غير تسجيل دخول.
- **التأثير:** تسريب بيانات طبية حساسة (HIPAA/GDPR-style risk) → فضيحة وفقدان ثقة فوري لو حصل تسريب.
- **الحل المقترح:** خلي الـ bucket private + استخدم `createSignedUrl()` بصلاحية محدودة (مثلاً 5 دقايق) بدل public URL، أو مرّر الطلب على Edge Function يتحقق من ownership الأول.

### C2 — حجز المواعيد العامة (Public Insert) ممكن يتزوّر عليه business_id/branch_id
- **الموقع:** `011_fix_all_rls.sql` — policy الـ INSERT على `appointments` لـ `anon` هي `WITH CHECK (true)` بدون أي تحقق إن الـ branch_id/business_id فعلاً مرتبطين ببعض أو إن العميل ده فعلاً بيحجز من صفحة الحجز الصحيحة.
- **التأثير:** أي حد يقدر يبعت request مباشر للـ API ويحجز/يزرع مواعيد وهمية في أي business لو عرف الـ branch_id بتاعه (IDs ممكن تتسرب من صفحات الحجز العامة).
- **الحل المقترح:** أضف تحقق في الـ policy أو trigger إن `branch_id` ينتمي لـ `business_id` المُرسل، وحط rate limiting (مثلاً عبر Edge Function أو Supabase Function) لمنع spam booking.

### C3 — باسورد الأدمن مكتوب في كود الفرونت (`AdminDashboard.jsx`)
- **الموقع:** `AdminDashboard.jsx:20` — `"MawidOwner@2025"` ثابت في الكود اللي بيتبني ويتشحن للمتصفح.
- **التأثير:** أي حد يفتح الـ JS bundle (حتى من dist/) يقدر يستخرج الباسورد ويفعّل/يعطّل أي حساب تجاري في النظام.
- **الحل المقترح:** الانتقال لـ Supabase Auth role-based admin (جدول `admins` + RLS بدل password ثابت)، أو على الأقل نقل الفحص لـ Edge Function بدل الفرونت.

### C4 — لا يوجد قفل حقيقي على السعة (Capacity) عند الحجز المتزامن
- **الموقع:** منطق الحجز بيعتمد على عدّ المواعيد المؤكدة في الفرونت قبل الإدراج، مفيش UNIQUE constraint أو DB-level lock.
- **التأثير:** لو حجزين دخلوا في نفس الثانية، الاتنين ممكن يعدّوا الفجوة فاضية ويحجزوا → over-booking فوق السعة المسموحة (خطير جدًا في العيادات اللي سعتها=1).
- **الحل المقترح:** أضف DB trigger/constraint يرفض الإدراج لو عدد confirmed appointments لنفس (branch_id, date, time) >= capacity.

### C5 — لا يوجد تحقق من أن تاريخ/وقت الموعد في الماضي
- **الموقع:** `appointmentSchema` (`src/lib/validators.js`) ما فيه أي قيد على إن `appointment_date` لازم يكون اليوم أو بعده.
- **التأثير:** يمكن حجز/تعديل مواعيد بتواريخ ماضية، يسبب لخبطة في التقارير والتذكيرات.

> **نصيحة:** خصص يوم كامل لإصلاح C1-C4 قبل أي اختبار قبول نهائي (UAT) أو إطلاق فعلي، لأن دول مش "bugs تجميلية" — دول data leak و overbooking وaccess control حقيقي.

## ترتيب الأولوية المقترح للتنفيذ

1. **أولًا (قبل أي إطلاق):** إصلاح C1 (ملفات طبية عامة)، C2 (تزوير business_id)، C3 (باسورد أدمن)، C4 (Race condition في الحجز) — هذول أخطر من أي باقي bug.
2. **ثانيًا:** اختبار المسارات الأساسية (Happy paths) لكل ميزة: تسجيل → onboarding → حجز → موعد → تقرير، من البداية للنهاية كرحلة مستخدم كاملة.
3. **ثالثًا:** اختبار الحالات الحدّية (Edge cases) المذكورة في كل قسم.
4. **رابعًا:** اختبار الأمان (قسم 11) والتحميل (13.4) والموبايل (قسم 12).
5. **خامسًا:** اختبار قبول من مستخدم حقيقي (Beta tester) على بيانات تجريبية وليست بيانات عملاء حقيقيين، لمدة أسبوع على الأقل قبل الإطلاق الكامل.
